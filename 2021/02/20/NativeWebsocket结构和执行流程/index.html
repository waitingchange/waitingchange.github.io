NativeWebsocket 结构和执行流程

梳理一下NativeWebsocket 的设计和执行流程，方便维护。

为什么会有NativeWebsocket ？

目前APP 中核心上课交互部分依赖Websocket，是嵌入在课件SDK 内部的Websocket，在这块儿我们收到比较多的报账投诉。鉴于JavaScript 版本的Websocket 对于Laya 引擎的强依赖和JavaScript 版本Websocket 在性能、功能和扩展性存在一些问题，我们需要使用原生版本的Websocket。通过原生版本Websocket，我们也可以对现有产品进行能力升级。

为什么不使用第三方提供的Websocket组件 ？

可以选用第三方已经稳定封装的组件来作为第一层socket 备份。使用原有接入第三方SDK 方式没有问题，能够达到目的。

分析一下现状，我们存在多个产品的APP，多个操作系统，使用第三方组件就会存在多端开发和维护的问题，对于业务扩展和维护都是一件不太容易的事情。

我们采用了C++ 作为底层组件的方案，好处就是功能开发只需要一份，外围接口和业务功能统一，开发和维护成本相对比较低，可以通过不同平台的编译器生成不同平台需要的执行文件。同样劣势也很明显，需要经过严格测试分析以后才能上线，而且无法通过热更新来解决，能减小影响范围的方式就是添加功能开关。

回归正题，讲一下这块儿的结构

如上图，单独摘出 Websocket 来说明。

根据现有产品现状，需要提供至少四端功能统一的SDK。

自下向上简单介绍：

SDK 部分

​ 关于SDK的基础框架，选择Boost 的原因在于，Boost 在C++ 有着大量的使用基础，而且相对比较稳定，使用Boost 可以减少大量的开发时间，劣势是Boost 整体相对庞大，但是好消息是可以通过模块儿分离方式能够优化二进制大小。

Websocket 库这里采用了以boost 为基础的 WebsocketCpp 做为基础基础网络库， log 系统可以使用Boost 提供的Log 日志库，也可以使用其他第三方库。

其余的扩展功能，可以以插件儿方式载入整体框架进行统一输出。

考虑到上层平台的不确定性，需要暴露扩展接口给到外围做接入。

  1. 第一个版本采用了“C”语言方式接口暴露，好处就是不需要考虑平台限制，“C”语言方式接口可以直接被外界使用。同样有弊端，可以依赖namespace 方式增加类似于类方式的结构，但依旧不属于类方式，也同样会污染内部的全局函数。

  2. 第二版本直接采用类方式扩展，划分为主动调用类和用于异步接受的代理类。优点在于属于面向对象的接口，便于用户理解和使用，弊端就是实现略微绕，需要考虑到不同平台的差异性。

外围接口部分

​ 因为产品历史性质，业务方强依赖Laya 引擎。SDK同时需要支持移动端、PC 端，也需要跟现有业务进行上下交互，不得不依赖不开源的Laya引擎，使用Laya封装的接口。对于移动端来讲C++ 跟iOS 和Android 交互已经是非常成熟的方式。

移动端

​ iOS 工程使用的Object-c可以粗暴的理解为就是C 语言，与C++进行数据交互不存在任何问题。

​ 作为移动端强大的阵营，也具备完善的Jni ，可以用不太麻烦的方式跟原生端Java 进行交互。

桌面端

​ 对于桌面端来讲，需要单独列出来说一下了。为了比较好的进行双端统一，使用了Electron引擎来进行相应开发，属于桌面端通用的做法。但是问题在于Electron 是基于Nodejs 进行再次扩展封装，Nodejs 的版本存在大量的版本，Nodejs 跟C++ 又不得不依赖Google V8。先看一下Nodejs 本身的结构。

  1. Node.js 标准库，这部分是由 Javascript 编写的，即我们使用过程中直接能调用的 API。在源码中的 lib 目录下可以看到。
  2. Node bindings，这一层是 Javascript 与底层 C/C++ 能够沟通的关键，前者通过 bindings 调用后者，相互交换数据。这一层是支撑 Node.js 运行的关键，由 C/C++ 实现。

现有可以跟Nodejs 交互的官方推荐方式为 Nan 、 Napi 。

​ Nan 全程：Native Abstraction for Node.js ，其表现上是一个Node.js包。依赖于node-gyp 进行编译，是其中一种方式，有多个版本，属于稳定分支，缺陷是如果Nodejs 版本变更，那么Nan 的一些API 可能会不一致，需要根据Nodejs 依赖版本重新修改代码再次编译。

​ NAPI 是从Nodejs 8.0 版本以后才具备的，N-API的引入是试图改善Nan 依赖node 版本需要重新修改代码的一种尝试。它跟底层JS引擎无关，只要N-API暴露的API足够稳定，那么node扩展的编写者就不用过分担忧node的升级问题。虽然当前属于稳定分支，但是对于交互部分一些API 依旧不够稳定。

这里存在一些问题，Nodejs官方文档 以文档为例，不管中英文，文档中对于API的介绍存在一些错误，具体是否能用需要看具体实际情况。

以上只是对Nodejs 做了简单介绍，中间一个很关键的内容是 Libuv ，JavaScript 是单线程语言，C++ 可以拥有多线程，C++ 中多线程异步跟JavaScript 通信需要依赖Libuv ，这也是Nodejs 之所以高效的原因。 Nodejs 的火爆是因为接口为JavaScript 语言，有着大量的开发人员，但是底层可以依赖C++ 提供高性能的稳定服。

JavaScript Interface

​ 因为具体业务执行在课件里面，需要做一个适配层用于业务方和下层原生层Websocket通信的内容，在适配层中会通过跟移动端和桌面端约定好的具体api 来进行具体业务。

讲一下NativeWebsocket 的结构

先看一下设计结构图

分为分为两块儿， 接口层 ，和真正执行的SDK 部分。

接口部分

接口分为两块儿，主动调用和异步数据回调。

主动调用部分有 创建、初始化、连接和断开。

回调接口主要有连接状态监控 ， on_open 、on_close , 和连接建立以后的on_message

SDK 结构

SDK 里面分为 wss 协议和ws协议连接。一个EndPointManager 用于跟接口通信和管理创建出来的Client 。每一个Client 可以创建多个connection ， 但是为了方便管理 ， 我们把connection 作为Client 的一个成员变量，也就是每一个Client 可以管理内在的一个connection。connection 通过 Boost 提供的Asio 网络库进行与服务器的connection 连接和数据交互。

因为考虑到C++ 内部需要在连接中断的时候自动重连，所以在Client 中存在一些重连的业务逻辑。设计为，当检测异常断线，会发起主动重连操作，直到重连成功，尝试次数为三次。同时当网络连接建立完成，为了保持connection 维持，在每一个Client 中都有一个心跳Timer。
